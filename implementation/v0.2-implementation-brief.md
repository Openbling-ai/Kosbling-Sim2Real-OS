# Kosbling Sim2Real OS — v0.2 Implementation Brief

Status: Draft implementation brief / next-phase target  
Date: 2026-03-24  
Audience: Team / implementation / product

---

## 1. Why v0.2 exists

v0.1 already proves that the project can function as:
- a CLI-first, shadow-first commerce runtime
- a multi-agent staged operating simulator
- a bounded 30-day chunked execution loop

But v0.1 also compresses too much of the real business journey.

The biggest product gap is:

> the simulator currently begins too close to “already operating”

The business docs describe a fuller path:
- select product
- evaluate supplier and MOQ
- choose procurement / shipping assumptions
- set price and margin targets
- prepare storefront assumptions
- prepare creative / influencer launch setup
- then enter operating simulation

v0.2 should close that gap.

---

## 2. One-line goal

> Extend the current runtime from a post-setup operating simulator into a fuller e-commerce studio simulator that explicitly models the pre-launch setup phase before sales begin.

Companion UI goal for this version:

> Expose the whole business through a CEO-first control surface so the user can understand the store as a single operating system, not only as a sequence of chunk logs.

---

## 3. v0.2 product promise

The user should feel like they are not only operating a store, but building a store from zero.

That means the system should simulate:
- whether the product is viable
- whether the sourcing setup makes sense
- whether the first inventory decision is sane
- whether the storefront / offer is launch-ready
- whether launch assets and creator strategy are in place
- only then whether the business starts selling

---

## 4. Core scope change from v0.1

v0.1 starts from:
- initialized shadow store state
- initialized product, inventory, pricing, and channel defaults

v0.2 should instead introduce an explicit **Setup / Launch Preparation** phase before the timed operating chunks.

Recommended v0.2 top-level loop:

1. idea intake
2. clarification
3. real grounding
4. setup simulation
5. launch readiness review
6. market snapshot / launch plan
7. day 1-5 operating chunk
8. later chunks as before
9. final battle report

This means v0.2 is not only:
- “add a setup phase”

It is also:
- make launch readiness a real runtime gate
- make blocker / accepted-risk handling explicit
- make the Web UI able to show the whole store from a CEO perspective

---

## 5. New phase: Setup / Launch Preparation

This is the highest-priority v0.2 addition.

The runtime should explicitly model at least these setup decisions:

### A. Product and offer definition
- product concept refinement
- target customer framing
- core value proposition
- pricing hypothesis
- margin target

### B. Supplier / sourcing assumptions
- supplier identity or supplier profile
- MOQ or first-order quantity
- unit cost range
- lead-time assumption
- procurement confidence / risk

### C. Logistics / fulfillment assumptions
- shipping mode
- landed-cost estimate
- packaging assumption
- fulfillment posture

### D. Storefront assumptions
- store status: draft / prepared / launched
- product listing readiness
- offer page readiness
- launch blockers

### E. Creative and launch preparation
- initial creative direction
- initial content / ad posture
- influencer / creator launch readiness
- whether launch assets are prepared enough to begin sales

At the end of setup, the system should be able to say:
- ready to launch
- not ready yet
- ready but risky

This is not just a UX label.
It should become a real runtime gate before operating chunks begin.

The setup phase should also leave an explicit decision trail:
- what assumptions were adopted
- what blockers were raised
- what blockers were resolved
- what risks were consciously accepted at launch

---

## 6. Agent implications

v0.2 should keep the existing team shape, but give each role explicit setup responsibilities.

### CEO
- coordinates setup readiness
- asks for missing launch-critical inputs
- decides whether the business can move from setup into launch
- owns the final operating picture shown to the user, including blocker acceptance and launch rationale

### Supply
- evaluates sourcing, MOQ, procurement, inventory, and shipping readiness

### Marketing / Growth
- evaluates whether there is enough launch demand signal
- prepares initial channel and testing posture

### Brand
- evaluates offer, creative direction, launch narrative, and listing readiness

### Finance
- validates margin, reserve cash, working-capital pressure, and first-buy risk

Execution agent remains in place, but in setup phase it should execute setup actions, not only operating actions.

---

## 7. State model extensions

v0.2 should extend state with explicit pre-launch fields instead of burying them inside defaults.

Recommended additions:
- `product_selection`
- `storefront`
- `launch_readiness`
- `supplier_profile`
- `launch_assets`
- `setup_decisions`

Examples:
- `storefront.status = draft | prepared | launched`
- `launch_readiness.status = blocked | risky | ready`
- `supplier_profile.name`
- `supplier_profile.moq`
- `launch_assets.creative_ready`
- `launch_assets.kol_ready`
- `launch_readiness.blockers`
- `launch_readiness.accepted_risks`

The operating simulation should only begin once these fields have been initialized deliberately.

---

## 8. Action surface additions

v0.2 likely needs a setup-oriented action layer in addition to the current operating actions.

Candidate additions:
- `set_supplier_profile`
- `set_initial_inventory_plan`
- `set_moq_plan`
- `prepare_storefront`
- `prepare_launch_assets`
- `set_launch_readiness`
- `delay_launch`
- `launch_store`

The exact names can evolve, but the conceptual boundary matters:
- setup actions prepare the business
- operating actions run the business

v0.2 should also strengthen canonical action integrity.

Observed v0.1 weakness:
- role summaries, CEO merge rationale, and final approved payloads can drift out of sync

So v0.2 should explicitly add:
- stronger post-merge action normalization
- payload validation against action intent
- consistency checks between:
  - action summary
  - merge rationale
  - approved canonical payload
- runtime warnings or hard failures when the canonical payload contradicts the merged decision

---

## 9. Artifacts to add

v0.2 should add at least two new artifact types:

### A. Setup report
- initial business assumptions
- sourcing assumptions
- launch blockers
- launch-ready / risky / blocked judgment

### B. Launch readiness report
- what is prepared
- what is missing
- what risk is being accepted if launch proceeds

### C. Setup blockers / readiness log
- which blockers were raised by which role
- which blockers were resolved
- which blockers were accepted as launch risks
- why the CEO allowed or delayed launch

These should appear before the first operating chunk.

Together, these setup artifacts should give the UI enough structure to show:
- the current store-building posture
- the launch gate decision
- the unresolved risks the CEO is accepting

---

## 10. Explicit non-goals for the first v0.2 slice

Do not turn v0.2 into a giant ERP or omnichannel simulator.

Still out of scope for the first v0.2 cut:
- full real supplier crawling stack
- real listing generation and publishing
- full live commerce execution coverage
- a large-scale GUI rewrite detached from the runtime roadmap

The point is to restore the missing business journey, not explode scope.

---

## 11. Web UI in v0.2

Web UI is in scope for v0.2, but it should follow the already prepared observability direction rather than inventing a separate product track.

Use these existing documents as the UI source of truth:
- `design/web-observatory-design.md`
- `implementation/web-ui-v0.1-brief.md`

How to interpret that UI scope inside v0.2:
- the runtime track should add the missing setup / launch-preparation phase
- the Web UI track should expose both setup-phase and operating-phase state, decisions, and causality
- the UI remains observer-first and replay-first, not a replacement for the runtime itself

The Web UI should also make setup blockers and launch-readiness gating visible, not only operating metrics.

The core UI question for v0.2 is:

> “If I were the CEO of this store, can I immediately understand whether the business is launchable, how it is operating, and what is most likely to break next?”

So the first UI should prioritize:
- whole-store operational visibility
- launch readiness and blocker visibility
- cash / inventory / offer / traffic / creative / sourcing posture in one place
- CEO decision rationale
- then chunk-by-chunk replay as a drill-down layer

The UI should not be treated as a generic replay reader.
It should feel like a CEO cockpit backed by the runtime.

---

## 12. Recommended implementation stance

This version should be executed as one coordinated v0.2 build, with internal feature tracks rather than disconnected product efforts.

Recommended tracks:

1. runtime setup and launch-readiness core
2. action governance and canonical payload integrity
3. setup artifacts and readiness logs
4. CEO-first Web UI foundation

---

## 13. Recommended implementation order

1. freeze and document v0.1 as post-setup operating runtime
2. add setup-state primitives
3. add setup-phase role prompts and setup actions
4. add launch-readiness gating before chunk 1
5. add blocker tracking and launch-risk acceptance logic
6. harden canonical action normalization and payload consistency checks
7. add setup artifacts
8. adapt the Web Observatory into a CEO-first whole-store cockpit
9. validate one full “idea -> setup -> launch -> chunk 1” path
10. validate that the UI can explain the store both at a global level and at a chunk level

---

## 14. Acceptance test for v0.2

The minimum v0.2 acceptance path should feel like this:

1. user gives idea and budget
2. system clarifies missing launch-critical facts
3. agents establish sourcing, inventory, margin, storefront, and creative assumptions
4. system emits a setup / launch-readiness report
5. user approves launch
6. only then does day 1-5 operating simulation begin
7. the Web UI can replay both the setup phase and the first operating chunk from the stored artifacts/state

Additional acceptance criteria:
- if a blocker such as missing supplier readiness is unresolved, the runtime can mark launch as `blocked` or `risky` instead of silently proceeding
- if CEO rationale says one thing but approved payload says another, the runtime catches that inconsistency before execution
- from the Web UI alone, the user can understand what the store is, whether it is ready to launch, what the top risks are, and why the CEO decided to launch or delay

If the runtime still jumps directly from idea to “already selling,” v0.2 is not complete.
