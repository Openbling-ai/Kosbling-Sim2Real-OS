## 4. One-line goal

> Build a local-first, shadow-only, chat-shaped OPC simulation MVP that can take one product idea, form a compact scenario, ground it lightly, run a staged 30-day simulation, persist state, and return convincing boss-facing updates — without real live integrations.

---

## 5. Non-negotiable product boundaries

### 5.1 What v0.1 / Layer 0 is
- IM-native in interaction shape
- built on top of OpenClaw’s existing chat/channel/session layer
- shadow-only
- local-first
- stateful
- staged
- user-controlled

### 5.2 What v0.1 / Layer 0 is not
- not a dashboard product
- not a new IM infrastructure layer
- not real Live Mode
- not real external commerce/ad execution
- not a full true multi-agent runtime yet
- not a giant market simulation platform

### 5.3 Critical interaction rule
The system must **not auto-continue** long simulation stages unless the user explicitly asks to continue.

## 6. First implementation slice

This is the exact slice to build first.

### 6.1 Required user journey
Support one credible end-to-end loop:

1. user gives a product/business idea in chat
2. system asks a very small number of clarifying questions if needed
3. system forms a compact scenario
4. system performs lightweight reality grounding
5. system returns a market/reality snapshot + recommendation framing
6. user chooses to start / adjust / switch
7. system runs a 30-day shadow simulation in staged 5-day chunks
8. after each chunk, system pauses and reports back in chat style
9. user may adjust between chunks
10. system completes the run and returns recap / battle report

That is the acceptance path.

### 6.2 Required implementation stance
For this first cut, it is acceptable for:
- one LLM to package outputs as multiple visible roles
- the internal execution logic to be simpler than the visible team layer
- multi-agent behavior to be partially simulated by structured role prompting rather than true distributed agent orchestration

## 7. Explicit in-scope capabilities

### A. Scenario formation
Implement a compact scenario object aligned with `references/scenario-spec.md`.

Need:
- scenario shape
- validation / normalization
- enough fields for one 30-day shadow run
- ability to form scenario from short user input + defaults

### B. Lightweight grounding
Implement a very lightweight grounding layer.

Need:
- first hard anchor: Google Trends
- support for a few grounded fields written into state / market snapshot
- lightweight web grounding hooks only if cheap and useful

Do **not** build a heavy research subsystem.

### C. State system
Implement a state model aligned with `references/state-model.md`.

Need:
- state initialization
- persistent latest state
- state updates during run
- logs sufficient for recap

### D. Runtime loop
Implement runtime aligned with `references/runtime-loop.md`.

Need:
- 30-day run
- 5-day chunking
- pause/continue behavior
- event injection in lightweight form
- chunk updates and final recap

### E. Action proposal layer
Implement structured action proposals aligned with `references/adapter-contract.md`.

Need:
- structured action proposal objects
- runtime application logic
- structured execution results

### F. Visible team packaging
Support a visible product layer that can present outputs through roles like:
- Kos
- Supply Chain Ops
- Social Media & Growth
- Brand & PR
- Finance Guard

This may be prompt-packaged rather than fully separate runtime agents in the first cut.

### G. Artifacts
Persist enough artifacts to support:
- latest state
- chunk logs / summaries
- final recap
- debugging basic run behavior

---

## 8. Explicitly out of scope

Do **not** implement in this first cut:
- real Shopify / Meta / platform writes
- real credentials or production integrations
- custom IM transport/channel layer
- heavy dashboard/UI
- full true multi-agent routing/runtime
- rich approval trees
- heavy calibration scoring systems
- complex branching timelines / parallel universes as a first-class engine feature
- giant profile libraries by category
- heavy competitor scraping pipelines
- cloud multi-tenant infra

You may leave clean stubs/interfaces where useful, but do not build these out.

---

## 9. Layer 0 product constraints that must survive engineering

The following product constraints must remain visible in the code and behavior:

### 9.1 Boss-facing style
The product should feel like an AI e-commerce team reporting upward to the boss.

### 9.2 Market snapshot before run
There should be a pre-run artifact / response that summarizes:
- market reality snapshot
- grounded assumptions
- initial recommendation framing
- user choice to start / adjust / switch

### 9.3 Staged chunk updates
The user should receive meaningful intermediate updates, not just a single final dump.

### 9.4 User-controlled progression
After each staged block, wait for the user.

### 9.5 Counterfactuals can be lightweight
A simple final “if you had done X differently” layer is enough.
This does not need to be a deep alternate-history engine in the first cut.

## 10. Suggested internal domains

Internally, it is sufficient to model a smaller enduring execution structure such as:
- marketing / ads
- supply chain
- finance

Brand/PR may initially be lighter or mostly present at the visible packaging layer, as long as the user-facing product shape still feels coherent.

Do not force a 1:1 mapping between visible personas and internal engineering modules if it makes the first cut heavier.

---

## 11. State requirements

The implementation must support at least these state domains:
- `meta`
- `product`
- `supply_chain`
- `marketing`
- `brand`
- `finance`
- `market_data`
- `decision_log`
- `events`

Use `references/state-model.md` as the reference shape.

Keep the state small enough to stay tractable.

---

## 12. Runtime requirements

The implementation must support:
- one 30-day shadow run
- staged 5-day reporting rhythm
- explicit user pause/continue
- simple event/risk injection
- final recap generation

The internal day loop can remain simple.

---

## 13. Engineering freedom

Codex may choose reasonable implementation patterns such as:
- Python package layout
- Pydantic/dataclass models or equivalent typed models
- JSON/YAML persistence strategy
- minimal service objects / reducers / state updaters
- simple prompt templates / role wrappers
- lightweight tests

Choose the simplest structure that preserves clarity and future extension.

## 15. Acceptance criteria

The first cut is successful if all of the following are true:

1. a user idea can become a compact scenario
2. Google Trends grounding can be pulled or mocked behind a clean boundary
3. a market snapshot can be produced before the run
4. a 30-day shadow run can complete
5. the run can surface 5-day staged updates
6. the run pauses for user input between stages
7. user adjustments can affect later stages
8. state persists cleanly enough for inspection/debugging
9. the final recap is understandable and decision-oriented
10. the implementation does not sneak in heavy out-of-scope systems

---

## 16. Preferred implementation attitude

When facing a tradeoff:
- prefer smaller over broader
- prefer clearer over more clever
- prefer one good end-to-end path over many half-built options
- prefer Shadow-first simplicity over future-proof overengineering

The goal is not to build the final platform.
The goal is to build the first believable, usable construction slice.
