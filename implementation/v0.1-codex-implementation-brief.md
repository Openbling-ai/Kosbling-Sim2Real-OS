## 3. Correct system framing

Do **not** implement this as a traditional simulator core with agents wrapped around it.

Implement it as:

> **an agent-native, commerce-specific, shadow-first runtime**

Meaning:
- agent cognition is central
- the commerce harness is the domain action surface
- state/world continuity is essential
- shadow execution is the current operating mode

---

## 4. One-line goal

> Build a local-first, shadow-first, chat-shaped, agent-native commerce runtime that can take one product idea, ground it lightly, maintain a continuous commerce world, let agents operate a commerce harness, and report staged results back to the user.

## 6. What to build first

Build the first bounded slice of the agent-native commerce runtime.

### Required path
1. user gives idea in chat
2. CEO agent interprets intent and asks minimal clarifying questions if needed
3. system forms compact scenario
4. system performs lightweight real grounding
5. system builds initial world/state
6. system emits market snapshot
7. user starts or adjusts
8. runtime proceeds in 5-day chunks across a 30-day shadow run
9. user can adjust naturally between chunks
10. system emits final battle report

That is the acceptance path.

## 7. Core architecture to implement

The first cut should explicitly contain these layers:

### A. Agent layer
At minimum:
- CEO agent behavior / coordinator logic
- role/domain-aware reasoning wrappers as needed
- role plans that can carry both action suggestions and concise watchouts
- CEO merge output that includes both a final action summary and merge rationale
- execution-agent behavior that commits approved actions through the active adapter

### B. Commerce harness layer
At minimum:
- marketing actions
- supply actions
- finance actions
- lightweight brand actions if cheap

### C. State/world layer
At minimum:
- world context
- commerce state
- decision log
- event log
- artifact pointers

### D. Grounding layer
At minimum:
- Google Trends provider boundary
- lightweight grounded market fields
- fallback only for dev/test/degraded mode

### E. Shadow execution layer
At minimum:
- structured action proposals
- proposal merge / approval
- execution agent + adapter boundary
- execution ordering for approved actions
- harness handlers
- shadow effects
- execution results
- LLM-first event generation with thin runtime validation/canonicalization

### F. Artifact layer
At minimum:
- market snapshot
- chunk updates
- final battle report
- required boss-facing metrics as defined in `references/derived-metrics.md`
- minimum artifact contents as defined in `references/artifact-contract.md`

Currently implemented and recommended for observability:
- a per-chunk team trace artifact recording role proposals, role watchouts, CEO merge rationale, execution order, and execution results

## 8. Grounding stance

The default product path should assume real networked grounding.

For v0.1:
- Google Trends is the first hard anchor
- lightweight web-assisted enrichment is allowed if cheap and useful
- offline fixture/mock data is allowed only for:
  - development
  - testing
  - degraded fallback

Do not let the default system drift into a disconnected toy mode.

---

## 9. Natural language stance

Do **not** add a separate user-facing adjustment DSL as a required first-class feature in v0.1.

Instead:
- the user speaks naturally
- the CEO agent interprets intent
- downstream structured action proposals are produced
- approved actions are committed through the execution boundary
- the active adapter determines whether the effects land in shadow or live execution

The structured boundary is between agent cognition and execution, not between the user and the system.

---

## 10. Commerce harness stance

Agents should not manipulate raw engineering internals directly.

They should operate through a commerce harness surface containing actions such as:
- adjust channel mix
- set budget
- pause ads
- launch creator campaign
- reuse creator creative
- adjust price
- reorder inventory
- change shipping mode
- switch to presale
- set cash reserve

The preferred Layer 0 flow is now:
- planner agents propose actions
- planner agents may also emit role-specific watchouts for CEO arbitration
- CEO or equivalent coordinator approves/merges them
- CEO explains the merge rationale
- an execution agent commits them through the active adapter
- the shadow harness applies the resulting world changes

The exact first list can stay compact.

## 11. State requirements

The implementation must preserve a continuous commerce world.

At minimum, state should cover:
- `meta`
- `world_context`
- `product`
- `supply_chain`
- `marketing`
- `brand`
- `finance`
- `market_data`
- `decision_log`
- `events`
- `artifacts`

Use `references/state-model.md` as the shape reference.

Layer 0 should implement the **canonical core variable set** from `references/state-model.md`, not a fully expanded business-world model.

### 11.1 Do not drive all state the same way
The first implementation should distinguish four update mechanisms:

1. **deterministic accounting**
   - revenue, cost, balance, profit, inventory quantities
2. **business constraints / harness rules**
   - reserve cash, shipping consequences, stock boundaries, paused ads
3. **grounded bounded uncertainty**
   - reach, CPC movement, creator performance, delay risk, trend-sensitive performance
4. **agent-driven judgment**
   - intent decomposition, prioritization, strategy adaptation

Codex may decide the engineering form, but should preserve this conceptual split.

---

## 12. Runtime requirements

The runtime must support:
- staged 30-day shadow run
- 5-day chunk reporting rhythm
- explicit pause/continue control
- lightweight grounding refresh
- structured proposal -> merge -> execution agent -> adapter -> shadow effect -> state update flow
- chunk-level observability for role proposals, merge rationale, and execution results

Do not overcomplicate the runtime with heavy orchestration in the first cut.

---

## 13. Explicitly out of scope

Do **not** implement in this first cut:
- broad real Shopify / Meta / ops write surfaces
- custom IM/channel infrastructure
- full generic multi-agent runtime platform work
- giant rules libraries
- giant profile libraries
- heavy competitor scraping pipeline
- full dashboard product
- deep branching simulation engine
- rich live-sync architecture

Stub boundaries are acceptable where helpful.
A narrow proof-of-plumbing live adapter may exist, but it is not part of the core v0.1 acceptance target.

## 15. Acceptance criteria

The first cut is successful if:
1. an idea can become a compact scenario
2. the CEO agent can drive the initial interaction
3. grounding can produce a credible market snapshot
4. a continuous commerce state can be initialized and updated
5. a 30-day shadow run can complete in staged chunks
6. user natural-language adjustments can affect later stages
7. the commerce harness mediates domain actions cleanly
8. team-level collaboration is observable through role plans, merge rationale, and execution results
9. the final battle report is understandable and useful
10. the implementation does not expand into out-of-scope platform work

---

## 16. Final instruction

Do not build a generic simulator with a chat shell.
Do not build a generic agent platform.

Build the first bounded slice of an **agent-native commerce runtime** that currently runs in **shadow mode**.
